PHX Compiler Architecture & Internals

A professional book-like reference guide to the structure, algorithms, and optimizations of the PHX Native PHP compiler runtime.

1. Introduction to PHX

What is PHX?

PHX is a modern native compiler and high-performance runtime for a subset of the PHP programming language, built from the ground up in Go. Rather than relying on traditional slow virtual machines or tree-walking evaluation, PHX implements a hybrid execution engine: a fast AST-walking development interpreter and a highly optimized transpiler that translates PHP code directly to native Go, which is then compiled to native machine binaries.

Why build a PHP-to-Go Compiler?

PHP is historically an interpreted scripting language. The standard Zend Engine parses PHP files to opcodes, executing them on a virtual machine. While modern JIT compilation in PHP 8 has improved CPU-bound performance, it still carries memory overhead, dynamic typing bottlenecks, and slow multithreaded concurrency. By compiling PHP directly to Go:

  • Native Speed: We leverage Go's advanced compiler optimizations and code generator, producing native machine executables.
  • Dynamic Concurrency: PHP is natively single-threaded. PHX introduces built-in channels and green threads (goroutines) through `spawn` and `channel` primitives.
  • Type Specialization: CPU-bound loops and operations are statically analyzed and compiled to raw machine integers, yielding 10x-20x speedups.

Core Architectural Components

The compiler is structured cleanly as a series of pipeline stages: 

[ Source Code (.php) ]
         │
         ▼  (Lexical Analysis)
[ Lexer / Tokens ]
         │
         ▼  (Syntax Analysis - Pratt Parser)
[ Abstract Syntax Tree (AST) ]
         │
         ├──► Option A: [ VM / Interpreter (Development) ]
         │
         └──► Option B: [ Transpiler / Code Generator ] ──► [ Go Compiler ] ──► [ Machine Code ]

2. Lexical Analysis (Lexer)

What is the Lexer?

The Lexer (or lexical analyzer) takes raw source code text as input and converts it into a stream of discrete, meaningful blocks called Tokens. For instance, the string $x = 42; is converted into tokens: [IDENT($x), ASSIGN(=), INT(42), SEMICOLON(;)].

Why do we need it?

Parsing raw strings directly is extremely error-prone due to layout differences (spaces, tabs, newlines, comments). The lexer removes all irrelevant spacing, tracks line numbers for error logging, and standardizes code structures so the parser only has to deal with strongly-typed tokens.

How is it implemented in PHX?

The lexer is defined in internal/lexer/lexer.go. It holds the source code string, keeps track of the current cursor position, and scans characters sequentially. Crucially, it supports mixed PHP/HTML content by starting in an HTML mode and transitioning into PHP mode upon encountering <?php.

Lexer State Machine Algorithm:
1. Read current character.
2. If in HTML mode, scan text until <?php. Emit an HTML token.
3. If in PHP mode, scan identifiers, numbers, and operators. Skip whitespaces and comments.
4. Transition back to HTML mode if ?> is parsed.

3. Syntax Analysis (Parser)

What is the Parser?

The Parser reads the stream of tokens produced by the lexer and constructs a hierarchical representation called the Abstract Syntax Tree (AST). It verifies that the code conforms to the grammatical rules of the language.

Why Pratt Parsing?

Traditional recursive descent parsers can struggle with operator precedence (e.g. ensuring 2 + 3 * 4 evaluates as 2 + (3 * 4) rather than (2 + 3) * 4). PHX solves this by using a Pratt Parser (also known as a Top-Down Operator Precedence parser).

How Pratt Parsing Works

Each token type is associated with two parse functions:

  • Prefix Parse Function: Invoked when a token appears at the beginning of an expression (e.g. negative signs -$x, group expressions (1 + 2), or identifiers).
  • Infix Parse Function: Invoked when a token appears between two expressions (e.g. addition +, multiplication *, or function calls ()).

Precedence levels are ordered sequentially:

Precedence LevelOperator / Token Type
LOWESTDefault level
EQUALS==, !=
LESSGREATER<, >, <=, >=
SUM+, -
PRODUCT*, /, %
PREFIXUnary operators like !, -
CALLFunction invocation $foo()
INDEXArray element access $arr[$key]

The Pratt Parsing Algorithm

func (p *Parser) parseExpression(precedence int) ast.Expression {
    prefix := p.prefixParseFns[p.curToken.Type]
    if prefix == nil {
        return nil
    }
    leftExp := prefix()

    for !p.peekTokenIs(token.SEMICOLON) && precedence < p.peekPrecedence() {
        infix := p.infixParseFns[p.peekToken.Type]
        if infix == nil {
            return leftExp
        }
        p.nextToken()
        leftExp = infix(leftExp)
    }
    return leftExp
}

4. Abstract Syntax Tree (AST)

What is the AST?

The AST is the final structural layout representing the source program. The nodes are split into two categories:

  • Statements: Code structures that perform an action but do not produce a value (e.g. if blocks, while loops, variable declarations).
  • Expressions: Code structures that evaluate to a value (e.g. arithmetic, string concatenation, function calls).

AST Node Implementation

Located in internal/ast/ast.go, all nodes implement the base Node interface, which provides formatting strings for debugging. Nodes are deeply nested: a FunctionStatement contains a list of Identifier parameters and a BlockStatement body containing other statement nodes.

5. The Virtual Machine / VM

What is the VM?

PHX contains a built-in virtual interpreter (VM) inside internal/vm/evaluator.go. It performs a tree-walking evaluation of the AST nodes.

How Scoping is tracked (Environments)

Variables and functions are stored in an Environment. Since PHP functions have local variable scopes that are separate from the global scope, environments can be nested: 

[ Global Environment ]
        ▲
        │ (Outer Reference)
[ Local Function Environment ]
If a variable is looked up inside a function, the VM checks the local environment. If not found, it resolves to outer scopes (e.g. static scope rules).

6. The AST-to-Go Transpiler

The Core Design Decision

Evaluating an AST recursively by walking the tree carries extreme runtime overhead (heap allocations, reflection, pointer chasing). To achieve high-speed compiled binary execution, PHX translates the AST directly to native Go source code.

The Zend-Style Value representation: Val

To preserve PHP's dynamic typing and runtime behaviors, the compiler includes a unified stack-allocated Go struct representing values: 

type Val struct {
    Type  int // 0: Nil, 1: Int, 2: Float, 3: Bool, 4: String, 5: Array, 8: Func, 9: Obj
    Int   int64
    Float float64
    Bool  bool
    Str   string
    Array []Val
    Chan  chan Val
    Func  func(args ...Val) Val
    Obj   Object
}
Operations on variables are implemented as native Go methods on this struct (e.g. Add(v1, v2), Lt(v1, v2), Eq(v1, v2)). This mimics the PHP Zend engine's zval structure but benefits from Go's compile-time optimizations.

Scoping and Closures: The `use` Statement

PHP uses explicit closure variable binding using the use keyword, e.g.: 

$x = 10;
$fn = function() use ($x) { return $x; };
Go's closures capture outer variables by reference, which can cause loop variable capture bugs where all workers see the final value of a loop variable. To guarantee correct PHP semantics in the compiled Go code, PHX wraps closure invocations inside an Immediately Invoked Function Expression (IIFE) in Go: 
func(v_x Val) Val {
    return Val{Type: 8, Func: func(args ...Val) Val {
        return v_x
    }}
}(v_x)
This captures the variable's value at the exact time of closure creation, preventing variable shadowing and capture issues.

7. Runtime & Concurrency

Goroutines as PHP Green Threads

PHX exposes Go's high-speed scheduler directly to PHP through the spawn(), channel(), send(), and receive() functions.

Thread Objects and Synchronization

When spawn(function) is called, the compiler generates a Go goroutine that runs the closure in the background. It returns a native ThreadObject. When the parent script needs to retrieve the thread's result, it calls the join() method. This suspends the parent execution until the background goroutine sends its final return value across an internal synchronization channel.

// PHP Concurrency
$t = spawn(function() {
    return 42;
});
$val = $t->join(); // Returns 42

8. Compiler Optimizations

The Type Specialization Engine

The primary reason the PHX compiler executes 10x faster than standard PHP is its static **Type Specialization** engine. Before compiling a math expression or loop, the compiler statically analyzes the expression tree using isIntExpr.

If the operands are detected as integer literals, loop counters, or integer-only variables, the code generator avoids wrapping the values inside the heavy Val struct and instead emits raw Go int64 code. For example, a loop like: 

for ($i = 0; $i < 1000; $i++) { $sum += $i; }
is translated directly to native Go: 
for v_i := int64(0); v_i < int64(1000); v_i++ {
    v_sum = NewInt(v_sum.Int + v_i)
}
This entirely removes heap allocations, memory jumps, and type check operations, allowing loops to execute at native hardware speeds.

Exception Recovery via Panics

PHP exceptions (throw new Exception()) are mapped directly to Go's panic/recover mechanism. The compiler wraps `try` statements in Go closures containing `defer recover()`. If an exception is thrown, it executes a Go panic containing the exception object. The recovery block catches this panic, extracts the exception instance, verifies if it matches the `catch` block class type, and executes the catch statement cleanly.

9. Step-by-Step Execution Flow

Here is what happens when you execute bin/phx run script.php:

  1. Read: The CLI reads script.php into memory.
  2. Tokenize: The Lexer scans characters and builds a Token stream.
  3. Parse: The Parser checks grammatical rules and constructs the AST.
  4. Compile: The Compiler traverses the AST and generates a corresponding Go file containing the runtime headers (`Val`, `NewInt`, arithmetic operators), target PHP script transpiled statements, and class structure tables.
  5. Write: The Go code is written to a temporary workspace under a unique directory name (e.g. phx_run_12345/main.go).
  6. Build: The CLI spawns the native Go compiler: go build -o temp_bin phx_run_12345/main.go.
  7. Execute: The compiled machine code executable is run, streaming output directly back to the terminal.
  8. Cleanup: Once finished, the temporary directory and binary are deleted automatically.
