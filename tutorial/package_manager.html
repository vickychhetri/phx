Package Manager Tutorial

Learn how to create, distribute, and consume reusable PHX modules using the built-in package manager CLI.

1. Introduction to PHX Package Manager

The PHX package manager (accessed via phx package or the shorthand phx pkg) provides native dependency management inspired by modern package ecosystems like Composer (PHP), NPM (Node.js), and Go Modules.

It handles automatic dependency resolution (resolving constraint satisfaction and conflict detection), locks package versions for deterministic builds via a lockfile, packages directories securely, and generates an autoloader that works in both interpretation and compiled standalone binary executions.

📦 Local Development Registry

By default, the package manager resolves packages from a local user directory at ~/.phx_registry/, allowing offline development and seamless package publishing without requiring external cloud accounts.

2. Initializing a Project

To define dependencies for your project, start by creating a phx.json manifest file. This acts as the configuration blueprint for your application package.

Interactive Setup

Run the package initialization command from your project root:

$ phx pkg init

This prompts you for the package name, initial version, description, and main entry file.

Non-Interactive (Default) Setup

To automatically skip the interactive questions and create a manifest with standard defaults:

$ phx pkg init -y

Manifest Structure (phx.json)

{
  "name": "my-application",
  "version": "1.0.0",
  "description": "High performance concurrent web service",
  "main": "index.php",
  "dependencies": {}
}

3. Managing Dependencies

Dependencies are packages from the registry required by your project. Adding them updates the manifest and downloads the target archives.

Adding a Dependency

Use the add command to register a package dependency:

$ phx pkg add logger

This automatically discovers the latest version, appends a caret constraint (e.g. ^1.0.0) to your phx.json, resolves the dependency chain, extracts the package files to phx_packages/, and creates the lockfile.

Specifying Version Constraints

You can define exact versions or wildcards for package installation:

Constraint Matches Example Command
^1.0.0 (Caret) Any version within the same major releases (>= 1.0.0, < 2.0.0). phx pkg add logger ^1.0.0
1.0.0 (Exact) Precisely version 1.0.0. phx pkg add logger 1.0.0
* or latest The absolute highest version published. phx pkg add logger *

Installing Dependencies (Deterministic Builds)

When deploying code or starting on a clean project clone, retrieve all requirements listed in phx.lock or phx.json by running:

$ phx pkg install

This creates a consistent local folder layout in phx_packages/.

Updating Dependencies

To upgrade packages to the latest matching constraints and regenerate your lockfile, run:

$ phx pkg update

4. Package Publishing

You can package your own custom modules and make them available to other projects on your machine. The publisher automatically zips files in your directory while excluding dependency caches like phx_packages/, .git/, or node_modules/.

Steps to Publish a Package

  • Create your package folder (e.g. packages_source/my-package).
  • Initialize a phx.json manifest detailing your package's metadata and dependencies.
  • Add your class files (e.g. src/Utility.php) and specify the main entry file in the "main" field of your manifest.
  • Run the publish command from within the package directory:
$ phx pkg publish

This zips the source directory and saves it to ~/.phx_registry/my-package/my-package-<version>.zip while registering the version details in the registry index.

5. Registry Search

Use the search utility to discover published packages available in the local registry.

List All Packages

$ phx pkg search

Filter Packages by Name

$ phx pkg search math

6. Autoloading Mechanics

When the package manager installs dependencies, it generates a single PHP routing file at phx_packages/autoload.php. This handles registering and importing every required class entry-point dynamically.

Using Autoloading in Your Code

At the start of your main program, simply include this file. You do not need to write multiple static include calls for each individual file anymore!

<?php
// Include all packages instantly
include "phx_packages/autoload.php";

$log = new Logger();
$log->info("Autoloader successfully initialized.");
?>

How it Works Under the Hood

PHX parses the include "phx_packages/autoload.php"; statement. During compilation (phx build), the static parser resolves these inclusions and incorporates the code definitions directly into the generated Go executable structure, resulting in a single native binary containing all package logic.

7. End-to-End Example

Let's create a complete workflow demonstrating the package manager. We will initialize a project, install the built-in logger and math-helper packages, and run a script using both classes.

Step 1: Setup Workspace & Initialize

$ mkdir test_project
$ cd test_project
$ phx pkg init -y

Step 2: Add Dependencies

$ phx pkg add logger
$ phx pkg add math-helper

Step 3: Write code (main.php)

<?php
include "phx_packages/autoload.php";

$logger = new Logger();
$math = new MathHelper();

$logger->info("Starting calculation...");
$result = $math->factorial(5);
$logger->success("Factorial of 5 is: " . $result);
?>

Step 4: Execute via Interpreter

$ phx run main.php
[INFO] Starting calculation...
[SUCCESS] Factorial of 5 is: 120

Step 5: Compile Standalone Machine Binary

$ phx build -o my_math_app main.php
$ ./my_math_app
[INFO] Starting calculation...
[SUCCESS] Factorial of 5 is: 120

8. Hosting Packages in the Cloud

The PHX package manager supports hosting and resolving packages in the cloud via static HTTP/HTTPS servers (such as GitHub, public filesystems, S3, or CDNs) for open-source distribution.

How to Publish to the Cloud

Because the registry structure is serverless and static, you can host your own remote package registry easily:

  • Publish your packages locally using phx pkg publish. This builds the package ZIPs and generates/updates the index files under your local ~/.phx_registry/ directory.
  • Create a public GitHub repository (e.g., github.com/username/phx-registry) or set up a static web server.
  • Push the contents of the ~/.phx_registry/ directory directly to the root of your repository or web server.

How to Consume Cloud Packages

To use packages from a remote cloud registry, configure the registry URL by setting the PHX_REGISTRY_URL environment variable when running commands:

$ PHX_REGISTRY_URL="https://raw.githubusercontent.com/username/phx-registry/main" phx pkg search
$ PHX_REGISTRY_URL="https://raw.githubusercontent.com/username/phx-registry/main" phx pkg install

The package manager will automatically fetch index.json metadata, download package ZIP archives, and cache them locally under ~/.phx_registry/cache/. Subsequent commands can run completely offline, utilizing the cached dependencies.

Direct Package URL Dependencies (Simplified Option)

Instead of setting up or configuring a static registry server, you can add dependencies directly using a repository URL or zip archive URL:

$ phx pkg add https://github.com/vickychhetri/package

This will download the repository, inspect its phx.json manifest to detect the package name, and register it directly in your phx.json manifest:

{
  "dependencies": {
    "package-name": "https://github.com/vickychhetri/package"
  }
}

When running phx pkg install, the package manager automatically fetches the package from the URL, handles local caching, and strips any top-level directory wrapper (e.g. GitHub folder wraps like package-main/) during extraction, keeping your phx_packages/ directory clean and ready to use.
