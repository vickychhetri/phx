Getting Started Tutorial

Build your first high-performance, concurrent, memory-cached REST API with PHX from absolute scratch.

1. Prerequisites & Environment

Before launching your first microservice, make sure the PHX compiler binary is built and available in your path.

Build the PHX Toolchain

Run the following command in the root folder of the PHX directory to build the compiler:

$ go build -o bin/phx ./cmd/phx

2. Creating a Basic Hello Server

Create a file named app.php in your working directory. We will instantiate the HttpServer class and listen on port 8085.

<?php
$server = new HttpServer();

$server->listen(8085, function($req, $res) {
    $res->status(200)
        ->header("Content-Type", "text/plain")
        ->end("Hello from PHX Cloud App!");
});
?>

Syntax Breakdown

  • new HttpServer(): Instantiates the enterprise HTTP server wrapper class.
  • $server->listen(8085, function($req, $res) { ... }): Binds the server to TCP port 8085 and registers a callback closure to execute for every incoming client connection.
  • $res->status(200)->header(...): Fluent response methods chaining to specify response attributes before sending.
  • ->end("..."): Ends the HTTP request transaction by writing the last chunk and flushing it to the network socket.

Underlying Meaning & Behavior

When compiled, the PHX compiler translates the HttpServer class directly to Go's standard net/http package. The callback closure is run inside an isolated Go goroutine spawned automatically for each client socket connection. This allows the server to scale concurrently to handle thousands of requests per second without blocking the main event execution loop.

Use-Case Scenario

Used to construct basic health-check endpoints, simple ping handlers, or webhook receivers that require minimal computation and rapid response delivery.

💡 Standard Response Chaining

The $res object supports method chaining on status() and header() to keep your code highly readable and clean.

3. Adding the Router & Parameters

For more complex APIs, register a Router instance. It compiles path patterns like /users/{id} to optimized regular expressions.

<?php
$server = new HttpServer();
$router = $server->router();

// Basic Route
$router->get("/", function($req, $res) {
    $res->status(200)->end("Home Route");
});

// Route with Dynamic Parameters
$router->get("/api/v1/users/{id}", function($req, $res, $id) {
    $res->json([
        "user_id" => $id,
        "name" => "User Account #" . $id,
        "roles" => ["developer"]
    ]);
});

$server->listen(8085);
?>

Syntax Breakdown

  • $server->router(): Resolves and returns the routing manager object.
  • $router->get("/pattern", function): Binds a GET HTTP request route.
  • {id}: A dynamic path parameter variable matching alphanumeric values in the URL.
  • $res->json(array): Formats and serializes a native array into a JSON payload, sets the Content-Type: application/json header, and sends it to the client.

Underlying Meaning & Behavior

The routing manager compiles URL patterns containing curly braces into Go regular expressions (e.g. ^/api/v1/users/([^/]+)$). When a request arrives, the router matches the path, parses the parameter arguments, and passes them dynamically to the callback closure arguments.

Use-Case Scenario

Developing RESTful microservices, API servers returning structured data from databases, or routing user resource requests based on unique identifiers.

4. Creating a Custom Middleware

Middleware allows intercepting requests before they reach the router. Middlewares receive the request, response, and a $next closure parameter. Calling $next($req) moves the request forward.

<?php
$server = new HttpServer();

// Logger Middleware
$server->use(function($req, $res, $next) {
    $start = microtime(true);
    
    // Pass execution down the pipeline
    $nextRes = $next($req);
    
    $elapsed = (microtime(true) - $start) * 1000;
    echo "[" . $req->getMethod() . "] " . $req->getPath() . " processed in " . $elapsed . "ms\n";
    return $nextRes;
});

// Guard Middleware
$server->use(function($req, $res, $next) {
    if ($req->getPath() === "/admin") {
        $headers = $req->getHeaders();
        if ($headers["Authorization"] !== "SecretToken") {
            $res->status(401)->json(["error" => "Unauthorized"]);
            return null; // Terminate execution early (do not call $next)
        }
    }
    return $next($req);
});

$server->listen(8085);
?>

Syntax Breakdown

  • $server->use(function($req, $res, $next) { ... }): Inserts a handler function into the global request processing stack.
  • $next($req): The execution step trigger callback. Initiates the next middleware or routes the request to its final destination.
  • return null;: Short-circuits the pipeline. Terminates request processing without letting downstream handlers see it.

Underlying Meaning & Behavior

Middlewares are modeled as a nested execution chain. When a middleware triggers $next($req), control yields to the subsequent function. When that function returns a response object, execution returns to the parent middleware, allowing it to inspect response structures, measure elapsed execution time, or compress payloads.

Use-Case Scenario

API authentication checks (Bearer token validation), rate limiting, request CORS headers initialization, audit logging, and metric tracking.

5. Adding In-Memory Cache

Boost your service's performance by storing slow operation results in the shared thread-safe Cache. The cache instance can be resolved via $server->cache() or new Cache().

<?php
$server = new HttpServer();
$router = $server->router();

$router->get("/api/data", function($req, $res) {
    $cache = new Cache();
    
    // Attempt retrieving from cache
    $data = $cache->get("my_slow_data");
    if ($data === null) {
        // Slow calculation or database lookup simulation
        $data = "High-performance result";
        
        // Cache the result for 30 seconds
        $cache->set("my_slow_data", $data, 30);
    }
    
    $res->json(["data" => $data]);
});

$server->listen(8085);
?>

Syntax Breakdown

  • new Cache(): Instantiates or references the shared in-memory Cache instance.
  • $cache->get("key"): Looks up a stored value. Returns null if key has expired or does not exist.
  • $cache->set("key", $val, $ttl): Stores a value with a Time-To-Live expiration limit in seconds.

Underlying Meaning & Behavior

The Cache maps string keys to values protected by a Go sync.RWMutex. It handles high-concurrency read and write cycles safely. A background cleaning worker checks keys and purges expired entries automatically, ensuring memory bounds are respected.

Use-Case Scenario

Caching third-party API payloads, storing configuration parameters, speed-optimizing database queries, and managing user session details.

6. Running & Testing Your Application

Run your PHP script directly with the PHX binary. The engine compiles it to a native Go server and launches it on port 8085:

$ bin/phx run app.php
PHX Server listening on http://localhost:8085

Test with curl

Test the dynamic parameters routing endpoint using curl:

$ curl -i http://localhost:8085/api/v1/users/99
HTTP/1.1 200 OK
Content-Type: application/json

{"user_id":"99","name":"User Account #99","roles":["developer"]}
